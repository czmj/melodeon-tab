# Roadmap

Steps in dependency order. Tick as completed; update this file as plans change.

- [x] **1. abcjs spike** — feed 2–3 real tunes (src/fixtures/*.abc) through abcjs, dump parsed structure. Confirm extraction of: pitch sequence + durations, bar positions, metre, key-signature-applied accidentals, ties, repeats. Decide: does MVP unroll repeats or tab as written? Output: findings note + decision, then the adapter shape. → **Done:** docs/abcjs-spike-findings.md; decision = tab as written, no unroll (ADR 0006).
- [x] **2. Core data model** — finalise types in src/domain: instrument schema, D/G treble instance (**pitch map currently a placeholder — fill in and verify against a real box before relying on it**), note sequence, fingering result (chosen candidate + alternatives + pinned flag + confidence). → **Done:** DG_STANDARD (verified 21-button D/G, lesterbailey.org); NoteEvent/Tune/Candidate/FingeredNote finalised; parseAbc emits Tune[]; spec + plan under docs/superpowers/. See ADR 0006.
- [x] **3. Candidate mapping + feasibility** — pure fn: note → candidates. Test against fixtures; confirm diatonic gaps surface correctly. → **Done:** src/engine/candidates.ts `mapTuneCandidates(tune, instrument)`; fixture tests confirm gaps surface (Jiggery's A# = unplayable). Cross-note bellows feasibility deferred to the DP (step 4).
- [x] **4. DP engine, naive costs** — constant reversal cost, small row-change cost. Signature `cost(from, to, context)` from day one, context ignored initially. Verify optimality and pin-and-recompute. → **Done:** src/engine/cost.ts `makeCostFn(instrument, weights)` (reversal 1, row-change 0.5; context threaded but unused); src/engine/fingering.ts `computeFingering(tune, lattice, cost, pins?)` — Viterbi with chain resets at rests/gaps, pin-and-recompute (ADR 0005). Tests verify greedy-beating optimality + pins. confidence/costMargin are placeholders until step 7.
- [x] **5. Ugly tab renderer** — text/bare HTML grid: button numbers, row marks, direction. First end-to-end moment: paste tune, see tab. → **Done:** src/render/tab.ts `renderTab(fingering, instrument)` → per-note cells (button number; ↑ red = push, ↓ blue = pull; underline = G row; `?` = unplayable; `-` = rest), pure view over FingeringResult (Invariant 5); App binds cells to styled spans. Wired into App above the debug table. Provisional format — real published tab conventions (Rennie/Mally/Biggs) verified in step 6.
- [ ] **6. Cost tuning** — decide tab style + verify its published conventions; hand-finger fixture tunes as ground truth; add beat-strength and phrase-boundary context; iterate weights until engine mostly agrees. Open-ended by design. Then: bellows-air management as DP coupling state (ADR 0007), sequenced **after** beat-strength context is wired — penalises long single-direction runs, breaking them on strong beats. → **In progress:** beat-strength wired into `cost()` (reversals beat-scaled, cheap onto strong beats) and bellows-air implemented as `(candidate, run)` DP state (ADR 0007) — long runs now break, reversals land on strong beats. Remaining: hand-tune the weights to taste. Bellows-air pressure (`airPenaltyRate`/`airComfortBeats` in `src/engine/cost.ts`) was rebased from a note-count run to a duration-based one (`CostContext.sameDirectionBeats`, accumulated from real `durationTicks`, not note onsets) — a run of quavers no longer costs the same as a run of whole notes at the same note count. Current defaults (`airPenaltyRate: 0.4`, `airComfortBeats: 8`) are a first pass, loosened further after early tuning felt too reversal-happy. Flagged as provisional — direction choice is hard to fully judge melody-only, since a real player picks bellows direction partly by what bass/chord they want sounding, which isn't modelled yet (post-MVP bass end). Mel Biggs' published convention is now verified and implemented in `TabCell` (ADR 0008); Rennie/Mally's remain unverified if picked up later. **Known gap:** `basicBeatStrength` (src/domain/notes.ts) has no concept of asymmetric-metre grouping — its only grouping-aware branch is `den === 8 && num % 3 === 0` (6/8-family jigs, dotted-quarter beats); every other metre falls back to treating each quarter-note beat as an equally-weighted "main beat" (0.6). That's wrong for irregular metres (5/4, 5/8, 7/8, 7/4…) which are felt as asymmetric sub-groups (e.g. 3+2), where the beat starting the second group should read stronger than a beat merely continuing one. Flagged via src/fixtures/tansys-golowan.abc (a 5/4 Cornish march) — not fixed yet; plain ABC `M:5/4` doesn't encode the 3+2 vs 2+3 grouping, so a fix needs a way to infer or default it. **Another tuning gap, scoped not built:** `cost()` has no pull toward a tune's diatonic home row (a G tune should mostly sit on the G row) — only `rowChange`, which penalises switching rows but has no notion of a "right" one. Design: docs/superpowers/specs/2026-07-11-home-row-preference-design.md — home key as per-row instrument data (reusable to a third row), matched by pitch class, plumbed in as a new `offHomeRow` cost term. Major keys only — "minor → relative major's row" was the first idea but is disproven by a real counter-example (Bear Dance, K:Em, played on the D row not the relative major's G row); minor/modal keys get no row preference at all rather than a rule known to be sometimes wrong.
- [x] **7. Interaction layer** — override popup, downstream recompute wiring, low-confidence marking, localStorage persistence. → **Done:** click a note → OverridePanel to pin a button (Map fed to computeFingering, recomputes downstream); low-confidence = near-tie margin (src/engine/confidence.ts, re-runs DP forcing best alternative; threshold 0.1), highlighted in the tab; pins + ABC persist to localStorage.

Post-MVP backlog (see ADRs + docs/melodeon-domain.md): bass end and chord suggestion, layout presets + editor, remaining tab styles, strategy presets UI, database/auth.
