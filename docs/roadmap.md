# Roadmap

Steps in dependency order. Tick as completed; update this file as plans change.

- [x] **1. abcjs spike** — feed 2–3 real tunes (src/fixtures/*.abc) through abcjs, dump parsed structure. Confirm extraction of: pitch sequence + durations, bar positions, metre, key-signature-applied accidentals, ties, repeats. Decide: does MVP unroll repeats or tab as written? Output: findings note + decision, then the adapter shape. → **Done:** docs/abcjs-spike-findings.md; decision = tab as written, no unroll (ADR 0006).
- [x] **2. Core data model** — finalise types in src/domain: instrument schema, D/G treble instance (**pitch map currently a placeholder — fill in and verify against a real box before relying on it**), note sequence, fingering result (chosen candidate + alternatives + pinned flag + confidence). → **Done:** DG_STANDARD (verified 21-button D/G, lesterbailey.org); NoteEvent/Tune/Candidate/FingeredNote finalised; parseAbc emits Tune[]; spec + plan under docs/superpowers/. See ADR 0006.
- [x] **3. Candidate mapping + feasibility** — pure fn: note → candidates. Test against fixtures; confirm diatonic gaps surface correctly. → **Done:** src/engine/candidates.ts `mapTuneCandidates(tune, instrument)`; fixture tests confirm gaps surface (Jiggery's A# = unplayable). Cross-note bellows feasibility deferred to the DP (step 4).
- [x] **4. DP engine, naive costs** — constant reversal cost, small row-change cost. Signature `cost(from, to, context)` from day one, context ignored initially. Verify optimality and pin-and-recompute. → **Done:** src/engine/cost.ts `makeCostFn(instrument, weights)` (reversal 1, row-change 0.5; context threaded but unused); src/engine/fingering.ts `computeFingering(tune, lattice, cost, pins?)` — Viterbi with chain resets at rests/gaps, pin-and-recompute (ADR 0005). Tests verify greedy-beating optimality + pins. confidence/costMargin are placeholders until step 7.
- [x] **5. Ugly tab renderer** — text/bare HTML grid: button numbers, row marks, direction. First end-to-end moment: paste tune, see tab. → **Done:** src/render/tab.ts `renderTab(fingering, instrument)` → per-note cells (button number; ↑ red = push, ↓ blue = pull; underline = G row; `?` = unplayable; `-` = rest), pure view over FingeringResult (Invariant 5); App binds cells to styled spans. Wired into App above the debug table. Provisional format — real published tab conventions (Rennie/Mally/Biggs) verified in step 6.
- [ ] **6. Cost tuning** — decide tab style + verify its published conventions; hand-finger fixture tunes as ground truth; add beat-strength and phrase-boundary context; iterate weights until engine mostly agrees. Open-ended by design. Then: bellows-air management as DP coupling state (ADR 0007), sequenced **after** beat-strength context is wired — penalises long single-direction runs, breaking them on strong beats.
- [ ] **7. Interaction layer** — override popup, downstream recompute wiring, low-confidence marking, localStorage persistence.

Post-MVP backlog (see ADRs + docs/melodeon-domain.md): bass end and chord suggestion, layout presets + editor, remaining tab styles, strategy presets UI, database/auth.
